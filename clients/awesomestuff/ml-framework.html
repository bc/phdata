Tenet What it means How we achieve it
Scalable Handle Black Friday without hiring. Don't let infrastructure become the bottleneck for new ML use cases. Stay portable if a vendor raises prices. Auto-scaling ECS (2→20 tasks) absorbs traffic spikes. One unified container pattern means one team operates all real-time models. Open-source stack (FastAPI, Redis, MLflow, Docker) runs on any cloud — no vendor lock-in.
Adaptable Iterate on models weekly. Expand to new use cases without rebuilding infrastructure. Integrate with existing systems without a rewrite. Same Dockerfile, different config — adding a 6th model is a YAML file, not a project. GitHub Actions deploys in minutes, not days. FastAPI endpoints plug into any system via HTTP — e-commerce platform, CRM, inventory, ad platforms.
Safe GDPR-compliant by design. Full audit trail. No data leaks through model artifacts or serving infrastructure. 72-hour deletion SLA cascading across Redis, warehouse, prediction logs, and downstream systems. MLflow tracks every version — model, dataset, hyperparameters, metrics. No PII in containers — models contain learned weights, not customer data.

The Framework (7 Dimensions)

For each model, evaluate:

# Dimension Why It Matters
1 Model & Algorithm Simplest effective approach — no over-engineering
2 Model Size & Inference Parameters, disk size, p50/p99 latency, CPU vs GPU
3 Feature Preprocessing How complex is the pipeline to get data into the model?
4 Peak Load & Serving QPS at Black Friday, scaling strategy
5 Response Time → Business Impact What latency is tolerable before it costs money?
6 GDPR Erasure What user data touches the model, and how hard is it to delete within 30 days?
7 Simplest Platform Data prep & training + Inference deployment — minimum moving parts

Applied: 5 Models

A. Fraud Detection (Real-Time)

Dimension Assessment Source
Algorithm XGBoost, 300-500 trees, max depth 6-8. Industry standard for tabular fraud. No deep learning needed. XGBoosting.com — XGBoost for Kaggle Credit Card Fraud Detection
Model size ~15-25 MB on disk. ~50-80 MB in memory. Trivial to serve. Medium — Methods to Boost XGBoost Model Inference Latency (2026)
Inference CPU only. p50 ~2-4 ms, p99 ~8-12 ms (model only). E2E with feature fetch: p50 ~15-30 ms, p99 ~50-80 ms. Medium — XGBoost Inference Latency Optimization
Features ~80-120 features. Velocity features (txns in last 1h/24h) are the hard part — requires Redis. Everything else is simple numerics + categoricals. Preprocessing: <1 ms if pre-cached.
Peak QPS 5,000-15,000 on Black Friday (8-15x normal). A single 8-vCPU node handles ~2-3K QPS. Need 5-8 instances at peak. Queue-it — 149 Black Friday Statistics (2026)
Medium — XGBoost Inference Latency Optimization
Response time Hard limit: 200 ms e2e (on checkout critical path). For reference, Stripe's fraud API (Stripe Radar) returns in ~100 ms. Each 100 ms added = ~1% conversion drop. At $500M revenue, that's ~$2M per 1%. False positive rate matters more than latency — target <0.5% FPR. ExecViva — Fraud KPIs: The Executive Guide
GDPR erasure HIGH complexity. User data in: training logs, feature store (Redis), prediction logs. Must cascade-delete from raw tables + feature snapshots + prediction audit log. Model itself doesn't need retraining (single user is negligible in millions). Fraud investigation hold (Art 17(3)(e)) allows retention for legal claims. GDPR-info.eu — Art. 17 Right to Erasure
Data prep & training EC2 dev instance (VSCode + SSH) for experimentation. Production training: GitHub Actions workflow_dispatch → spins up EC2 runner, pulls data from warehouse, trains XGBoost, exports ONNX via onnxmltools, packages model.tar.gz, uploads to S3. Tracked by MLflow. Scheduled weekly retrain via cron trigger.
Inference deployment GitHub Actions workflow_dispatch → builds Docker image (FastAPI + ONNX Runtime), pushes to ECR, registers new ECS task definition with MODEL_ARTIFACT_S3 env var, updates ECS Fargate service. Container startup: entrypoint.sh downloads model.tar.gz from S3, extracts to /opt/ml/model. ONNX Runtime serves XGBoost at ~2-4x native speed. ALB health checks on /ping. Autoscaling 2-20 tasks. Same container pattern as all real-time models.

B. Recommendation Engine (Real-Time)

Dimension Assessment Source
Algorithm Two-stage: (1) ALS matrix factorization for candidate retrieval, (2) Optional XGBoost ranker for re-ranking top 50. Skip deep learning (two-tower) unless you have a dedicated ML team. Meta Engineering — Recommendation Systems at Facebook
Model size ALS: User matrix 1M users × 128 factors = ~512 MB (but only look up 1 vector at inference = 512 bytes). Item index (FAISS): ~35 MB. Ranker: ~8 MB. FAISS Wiki — Vector Codec Benchmarks
Inference FAISS ANN retrieval: p50 ~5-10 ms, p99 ~20-30 ms. With ranking: add 2-4 ms. E2E: p50 ~10-15 ms, p99 ~30-50 ms. CPU is fine at this scale. GitHub — facebook/faiss: Efficient Similarity Search
Features Training: implicit feedback matrix (views, carts, purchases) — sparse matrix, low preprocessing. Inference: single user vector lookup + ANN search. Cold start: fall back to category popularity. Preprocessing complexity: LOW at inference, MEDIUM at training.
Peak QPS 20,000-50,000 on Black Friday (fires on every page load). BUT: precompute top-20 recs for active users nightly → ~70-80% cache hit rate → real-time QPS drops to 5-15K. Algolia — BFCM 2025: Peak Demand, Perfect Performance
Response time Target: <100 ms. Often loaded async, so 200-300 ms is tolerable without conversion impact. Quality matters more than speed — 10% relevance improvement = 2-5% GMV uplift. Barilliance — Personalized Product Recommendation Statistics
GDPR erasure MEDIUM-HIGH. Must delete: interaction logs, user embedding vector (Redis key delete), cached recs, impression/A/B test logs. Model: delete user's row from the ALS user matrix; full retraining on next weekly cycle naturally excludes them. Cross-system propagation to recommendation cache is the real work.
Data prep & training EC2 r6i.4xlarge (128 GB RAM) for ALS training via implicit library. GitHub Actions triggers training run, pulls interaction data from warehouse, trains ALS factors, builds FAISS IVF index, packages model.tar.gz (user/item factors + FAISS index) → uploads to S3. Precomputes nightly recs for top 100K users → writes to Redis. Tracked by MLflow.
Inference deployment Same Docker/ECS pattern as fraud. Container loads FAISS index + user factors from S3 artifact at startup. FastAPI /predict does in-process FAISS ANN lookup (~5 ms). FAISS is already optimized C++ — no ONNX needed. Redis caches precomputed recs for known users. ECS Fargate behind ALB, autoscaling 2-15 tasks. Same Dockerfile, different MODEL_TYPE=als_faiss.

C. Inventory / Demand Forecasting (Batch)

Dimension Assessment Source
Algorithm LightGBM global model (one model for all SKUs). 2,000-5,000 trees, ~200 features. Regression target: units sold in next 7/14/28 days. Beats Prophet on accuracy for e-commerce. Fallback for new SKUs: category-level moving average. ScienceDirect — M5 Accuracy Competition: Results, Findings, and Conclusions
Model size 60-80 MB on disk. 120-300 MB in memory. Tiny by any standard.
Inference Batch only. Score 100K SKUs in ~30-120 seconds on a single CPU. Full pipeline (feature gen + scoring + write): 30-90 min.
Features ~150-250 features per (SKU, date). Lag features (sales 7d/14d/28d/1yr ago), rolling stats (7d/28d/90d mean/std/min/max), calendar features, promo flags, category, price tier. Preprocessing complexity: MEDIUM-HIGH — lag/rolling features require careful time-based windowing to avoid data leakage. Feature engineering is 60% of the work.
Peak QPS N/A — batch. Daily job: 100K SKUs. Weekly deep run: 500K+ SKUs. A single 16-vCPU / 64 GB node handles this easily.
Response time Batch SLA: results by 6 AM daily (before buying team starts). Missing the forecast → suboptimal replenishment → stockouts. 1% stockout rate on $500M GMV = ~$5M lost sales. Mirakl — The $1.2T Problem: Why Out-of-Stocks Are Crushing Retailers
GDPR erasure LOW. SKU-level model, not user-level. Training data is aggregate sales counts per SKU per day — no individual user data. Deleting a user's orders from raw data could change aggregates, but negligibly at scale. No model retraining needed.
Data prep & training Feature engineering in SQL (warehouse-side). GitHub Actions triggers LightGBM training on EC2, pulls features as Parquet from S3. Exports ONNX model via onnxmltools.convert_lightgbm() for validation. Packages artifact → S3. Tracked by MLflow. Scheduled via GitHub Actions cron (schedule: trigger).
Inference deployment No containers needed — batch only. Trained model runs as a scheduled Python script (GitHub Actions cron) that loads the S3 artifact, scores all SKUs, and produces a ranked operations digest — a prioritized purchasing plan that respects capital constraints and flags storage-fee risk. Operations opens a single report each morning and knows exactly what to order.

D. Customer Segmentation (Batch)

Dimension Assessment Source
Algorithm K-Means on standardized RFM + behavioral features, K=8-15 clusters. PCA (50→15 components) before clustering handles correlation/scale. GMMs give soft assignments if needed. Skip DBSCAN — arbitrary shapes don't map to marketing personas. ResearchGate — Customer Segmentation Using RFM and K-Means Clustering
Model size <1 MB. Centroids: 10 clusters × 50 features × 4 bytes = 2 KB. PCA matrix: 3 KB. Scaler: 400 bytes. The model is trivially small — it's just centroid coordinates.
Inference Batch. Assign 1M customers in <60 seconds (distance calculation). Full pipeline: 1-4 hours (warehouse query time dominates).
Features ~30-80 features. RFM core (recency, frequency, monetary), category preferences (% spend in 10-20 cats), device type, channel responsiveness, return rate. Must StandardScale all features (K-Means is distance-based). Log-transform monetary features. Preprocessing complexity: LOW-MEDIUM.
Peak QPS N/A — batch weekly/monthly. Output (user→segment) may be queried real-time as a simple Redis key-value lookup (~1 ms).
Response time Batch SLA: before next campaign send (T+24h is fine). Marketing is not latency-sensitive. Missing a weekly re-segmentation by a day has minimal impact.
GDPR erasure MEDIUM. Must delete: user's behavioral features, cluster assignment (user_id→segment_id). Model itself: centroids are group statistics — single user deletion doesn't change them. Cross-system propagation is the real challenge — segment assignments flow to CRM (Salesforce), customer engagement (Braze — a customer engagement platform for email, push, SMS), ad platforms. All must receive the deletion signal.
Data prep & training GitHub Actions daily schedule (4 AM UTC) + manual workflow_dispatch. Builds Docker container → pushes to ECR. RFM feature engineering in SQL, K-Means + PCA in scikit-learn. Model artifact (centroids + scaler + PCA) saved to S3. Tracked by MLflow.
Inference deployment GitHub Actions kicks off a one-shot ECS Fargate task (not a long-running service). Container runs the full pipeline: munge → train → score → generate HTML report → upload everything to S3. Outputs: report.html (shareable with marketing/sales via presigned URL), model.tar.gz, metrics.json. Also writes to segmentation/latest/ for stable URLs. Marketing receives the presigned link and is responsible for enacting campaigns on Braze, Salesforce, ad platforms, etc.

E. Dynamic Pricing (Batch)

Dimension Assessment Source
Algorithm Hybrid: ML demand elasticity (batch) + rule engine (real-time). Offline: LightGBM regression trained on historical price-demand data → outputs price elasticity per SKU. Online: Rule engine applies elasticity + constraints (price floor, ceiling, MAP, competitor parity). Optional: Thompson Sampling bandit for price point A/B testing. Do NOT start with pure RL — business constraints make it impractical. ScienceDirect — Dynamic Pricing with ML: Survey and Practice
Model size Elasticity LightGBM: 20-50 MB. Per-SKU coefficient table: ~8 MB in Redis (100K SKUs × 10 coefficients). Rule engine: stateless code, no model.
Inference Batch. Repricing worker runs every 10 minutes as a scheduled ECS task. Scores all SKUs (or priority subset), applies rule engine, writes final prices to Redis. Backend reads pre-computed price from Redis in 1-3 ms. Prices are at most ~10 minutes stale — acceptable for e-commerce.
Features ~80-120 for elasticity model. Demand velocity (rolling sales in 1h/6h/24h — same Redis pattern as fraud), inventory level, competitor price (requires scraping pipeline), promo flags, seasonality. Preprocessing complexity: MEDIUM. Same Redis velocity pattern as fraud — if you build it for fraud, reuse it here.
Peak QPS N/A — batch repricing. Price reads from Redis can handle 15-40K QPS on Black Friday, but the repricing worker itself processes ~1-5K SKUs/cycle every 10 min. Queue-it — 149 Black Friday Statistics (2026)
Response time Batch SLA: prices refreshed every ≤10 minutes. Backend reads from Redis (~1-3 ms), so page load is never bottlenecked by pricing. 10-minute staleness is acceptable — consistency matters more than freshness: "price shock" (different price shown vs checkout) is the #1 driver of cart abandonment. The price in Redis is the price at checkout — no discrepancy. Baymard Institute — Cart Abandonment Rate Statistics (2026)
GDPR erasure LOW-MEDIUM. Elasticity model is SKU-level, not user-level — no user data in the model. Must delete: user session/event logs (if demand velocity uses per-user signals), pricing impression logs (user_id saw price $X). No model retraining needed.
Data prep & training Monthly retrain: GitHub Actions triggers LightGBM training on EC2, exports ONNX via onnxmltools, packages model.tar.gz → S3. Every-10-min repricing: GitHub Actions cron triggers batch scoring of all SKUs, writes updated prices to Redis. Tracked by MLflow.
Inference deployment Two-layer architecture: pre-computed prices in Redis + background repricing worker. The e-commerce backend never calls the ML model directly — it reads the latest price from ElastiCache Redis (1-3 ms GET). A background repricing worker (ECS scheduled task) runs every 10 minutes, re-scores SKUs through the ONNX model + rule engine, fetches competitor prices (e.g. Prisync API), and writes updated prices back to Redis. Amazon adjusts prices ~2.5 million times per day — the average product price changes every ~10 minutes somewhere on the site. Repricing strategy: if the full catalog (100K SKUs) reprices in <10 minutes, just run them all every cycle. If it takes longer, set up a priority queue ranked by pageview activity — high-traffic SKUs reprice every cycle, long-tail SKUs reprice hourly/daily. CamelCamelCamel — Amazon Price Tracker

Design Principles

Principle What It Means
No vendor lock-in Every component is open-source or has drop-in replacements. FastAPI, Redis, MLflow, GitHub Actions, Docker — all run anywhere. Swap your cloud provider without changing application code.
Shared base, per-model code All models share the same base Docker image (FastAPI + ONNX Runtime + common libs). Each model has its own Python module for data prep, feature queries, and database connectors — fraud pulls velocity features from Redis, reco queries FAISS + user vectors, pricing fetches competitor prices and inventory levels, etc. MODEL_TYPE env var selects which module runs. Same Dockerfile, different entrypoint logic.
Redis is the shared brain One Redis cluster serves all models that need fast lookups: velocity features (fraud), user embeddings + rec cache (reco), pre-computed prices (pricing). Build the Redis infrastructure once.
Batch is just a script Forecast, Segmentation, and Pricing don't need real-time serving — they're GitHub Actions cron jobs. Pricing runs every 10 min, the others daily. Don't over-engineer what a schedule: trigger can handle.
MLflow is the registry Every model — from the trivial K-Means to the production XGBoost — registers in MLflow with versioning, metrics, and artifacts. One place to find any model.

Shared Infrastructure (build once)

Component Role Used By Cloud-Agnostic Options
Redis Feature store, cache, embeddings Fraud, Reco, Pricing Any managed Redis (ElastiCache, Memorystore, Azure Cache, Aiven) or self-hosted. Valkey as OSS drop-in.
FastAPI + ONNX Runtime + Docker Real-time model serving (optimized) Fraud, Reco (real-time); Pricing (batch worker reuses same image) ECS Fargate behind ALB. ONNX Runtime gives 2-4x speedup on tree models (XGBoost, LightGBM). FAISS stays native. Container portable to any runtime: K8s, Cloud Run, Fly.io.

Data warehouse Training data + batch outputs All 5 Snowflake, BigQuery, Redshift, DuckDB, Postgres. The models don't care — they read DataFrames.
GitHub Actions cron Batch orchestration Forecast, Segmentation, Pricing (all batch) GitHub Actions schedule: trigger handles all batch jobs — no separate orchestrator needed. If outgrowing GH Actions, migrate to Prefect, Dagster, or Airflow. All are open-source.
GDPR deletion pipeline Cascade user data deletion All 5 GitHub Actions workflow or any workflow engine. Fan out to Redis, warehouse, object store, CRM. No cloud-specific service required.

GDPR Deletion Architecture

Deletion Request (user_id)
    │
    ├── Redis: DEL user:{id}:* (features, embeddings, cached recs, coefficients)
    ├── Warehouse: DELETE FROM ... WHERE user_id = ? (training data, feature tables)
    ├── Object store: Mark for deletion in data catalog (training snapshots)
    ├── Prediction logs: DELETE WHERE user_id = ? (audit/monitoring tables)
    ├── Downstream: Push deletion to CRM, email, ad platforms via reverse ETL
    │
    └── Model retraining: NOT required per-request
        (next scheduled retrain naturally excludes deleted data)

SLA: All deletions complete within 72 hours of request
GDPR deadline: 30 calendar days
Buffer: 27 days for edge cases, manual review, legal holds

Deployment Order (Risk-Adjusted)

Order Model Rationale
1 Segmentation Lowest risk, batch, <1 MB model, no latency. Quick win for marketing. Validates warehouse → CRM pipeline.
2 Forecasting Batch, no user PII, no latency. Validates GH Actions cron + LightGBM + MLflow pattern.
3 Fraud First real-time model. Highest business value. Deploy in shadow mode first (score but don't block). Validates Redis + FastAPI + Docker pattern that all real-time models reuse.
4 Pricing Batch repricing worker reuses fraud's Docker + Redis infrastructure. Rule engine needs business stakeholder alignment (takes time). 10-min cycle is non-critical — safe to iterate.
5 Reco Highest QPS, most complex caching. Deploy last so infrastructure is battle-tested.
