Tenet What it means How we achieve it
Scalable Handle Black Friday without hiring. Don't let infrastructure become the bottleneck for new ML use cases. Stay portable if a vendor raises prices. Auto-scaling ECS (2→20 tasks) absorbs traffic spikes. One unified container pattern means one team operates all real-time models. Open-source stack (FastAPI, Redis, MLflow, Docker) runs on any cloud — no vendor lock-in.
Adaptable Iterate on models weekly. Expand to new use cases without rebuilding infrastructure. Integrate with existing systems without a rewrite. Same Dockerfile, different config — adding a 6th model is a YAML file, not a project. GitHub Actions deploys in minutes, not days. FastAPI endpoints plug into any system via HTTP — e-commerce platform, CRM, inventory, ad platforms.
Safe GDPR-compliant by design. Full audit trail. No data leaks through model artifacts or serving infrastructure. 72-hour deletion SLA cascading across Redis, warehouse, prediction logs, and downstream systems. MLflow tracks every version — model, dataset, hyperparameters, metrics. No PII in containers — models contain learned weights, not customer data.

The Framework (7 Dimensions)

For each model, evaluate:

# Dimension Why It Matters
1 Model & Algorithm Simplest effective approach — no over-engineering
2 Model Size & Inference Parameters, disk size, p50/p99 latency, CPU vs GPU
3 Feature Preprocessing How complex is the pipeline to get data into the model?
4 Peak Load & Serving QPS at Black Friday, scaling strategy
5 Response Time → Business Impact What latency is tolerable before it costs money?
6 GDPR Erasure What user data touches the model, and how hard is it to delete within 30 days?
7 Simplest Platform Data prep & training + Inference deployment — minimum moving parts

A. Fraud Detection (Real-Time)

Dimension Assessment
Algorithm XGBoost, 300-500 trees, max depth 6-8. Industry standard for tabular fraud.Supported
"Best ROC AUC of 0.984 on Kaggle fraud dataset with max_depth=5 and n_estimators=300. Ensemble methods like XGBoost have demonstrated significant promise for tabular fraud detection."
XGBoosting.com — XGBoost for Kaggle Credit Card Fraud Detection
No deep learning needed.
Model size ~15-25 MB on disk. ~50-80 MB in memory.Supported
"A 500-tree XGBoost model with depth 6-8 typically serializes to 10-30 MB (binary format). In-memory footprint is 2-4x disk size due to tree node structures and metadata."
Medium — Methods to Boost XGBoost Model Inference Latency (2026)
Trivial to serve.
Inference CPU only. p50 ~2-4 ms, p99 ~8-12 ms (model only)Supported
"You can achieve approximately 1ms latency by fixing threading issues. Restrict the booster to 1 thread for single predictions to reduce overhead." With 500 trees and batching, 2-4ms p50 is realistic.
Medium — XGBoost Inference Latency Optimization
. E2E with feature fetch: p50 ~15-30 ms, p99 ~50-80 ms.
Features ~80-120 features. Velocity features (txns in last 1h/24h) are the hard part — requires Redis. Everything else is simple numerics + categoricals. Preprocessing: <1 ms if pre-cached.
Peak QPS 5,000-15,000 on Black Friday (8-15x normal)Partial — traffic spike varies
SDE Ray: "It's not unusual for retailers to see 5 to 10x normal traffic during Black Friday and Cyber Monday." Queue-it: "Black Friday web visits were +78% vs average October Friday" (overall ~2x). The 8-15x multiplier is aggressive — 5-10x is better documented. For individual merchants during flash sales, spikes can exceed 10x. The 5-15K QPS range assumes a $500M GMV merchant processing ~600-1,000 orders/min at peak.
Queue-it — 149 Black Friday Statistics (2026)
. A single 8-vCPU node handles ~2-3K QPSPartial — depends on model size
XGBoost single-row inference is ~1ms with optimized threading. An 8 vCPU node (any cloud) handling 2-3K predictions/sec is consistent with ~1ms/prediction × 8 cores with overhead. Intel daal4py can provide up to 36x speedup. Exact throughput varies with tree count and depth.
Medium — XGBoost Inference Latency Optimization
. Need 5-8 instances at peak.
Response time Hard limit: 200 ms e2e (on checkout critical path). Each 100 ms added = ~1% conversion dropSupported
Greg Linden (Amazon, 2006): "In A/B tests, Amazon tried delaying the page in increments of 100 milliseconds and found that even very small delays would result in substantial and costly drops in revenue." Note: original finding was about sales/revenue, not conversion rate specifically. A 2017 Akamai study found the impact may be even larger.
GigaSpaces — Amazon: Every 100ms of Latency Cost 1% in Sales
. At $500M revenue, that's ~$2M per 1%. False positive rate matters more than latency — target <0.5% FPRNeeds context
Industry data: "Average merchants decline 2.6% of all orders for suspected fraud, and 30-70% of those are false positives" — meaning false decline rates are ~0.8-1.8% of total orders. A <0.5% FPR target (fewer than 1 in 200 legitimate transactions blocked) is aggressive but achievable with modern ML. Note: this metric differs from alert-level FPR used in AML (30-50%).
ExecViva — Fraud KPIs: The Executive Guide
.
GDPR erasure HIGH complexity. User data in: training logs, feature store (Redis), prediction logs. Must cascade-delete from raw tables + feature snapshots + prediction audit log. Model itself doesn't need retraining (single user is negligible in millions). Fraud investigation hold (Art 17(3)(e)) allows retention for legal claimsPartial — no fixed period
Art 17(3)(e): "The right to erasure does not apply where processing is necessary for the establishment, exercise or defence of legal claims." This is correct — but GDPR does not specify a fixed 12-18 month period. The retention period must be justified by the specific legal claim timeline (e.g., statute of limitations).

Production training: GitHub Actions workflow_dispatch → spins up EC2 runner, pulls data from warehouse, trains XGBoost, exports ONNX via onnxmltools, packages model.tar.gz, uploads to S3. Tracked by MLflow. Scheduled weekly retrain via cron trigger.

Container startup: entrypoint.sh downloads model.tar.gz from S3, extracts to /opt/ml/model. ONNX Runtime serves XGBoost at ~2-4x native speed. ALB health checks on /ping. Autoscaling 2-20 tasks. Same container pattern as all real-time models.

B. Recommendation Engine (Real-Time)

Dimension Assessment
Algorithm Two-stage: (1) ALS matrix factorization for candidate retrieval, (2) Optional XGBoost ranker for re-ranking top 50Supported
The two-stage retrieval + ranking architecture is the industry standard pattern used at Meta, YouTube, and Pinterest. ALS is a proven baseline for implicit feedback. Two-tower models require dedicated ML infrastructure and team investment.
Meta Engineering — Recommendation Systems at Facebook
. Skip deep learning (two-tower) unless you have a dedicated ML team.
Model size ALS: User matrix 1M users × 128 factors = ~512 MBSupported — math checks out
1M users × 128 floats × 4 bytes/float = 512 MB. This is standard dimensionality for collaborative filtering (Spotify uses 40-256 factors, Netflix Prize solutions used 50-200). FAISS IVF index for 100K items × 128d ≈ 35 MB is consistent with FAISS codec benchmarks.
FAISS Wiki — Vector Codec Benchmarks
(but only look up 1 vector at inference = 512 bytes). Item index (FAISS): ~35 MB. Ranker: ~8 MB.
Inference FAISS ANN retrieval: p50 ~5-10 ms, p99 ~20-30 msSupported
FAISS benchmarks show HNSW "maintains near-perfect accuracy while drastically improving query speed" and IVF-Flat "compresses the search space to achieve sub-millisecond responses." For 1M vectors with 128 dimensions, typical CPU query times are a few milliseconds per lookup. 5-10ms p50 is conservative.
GitHub — facebook/faiss: Efficient Similarity Search
. With ranking: add 2-4 ms. E2E: p50 ~10-15 ms, p99 ~30-50 ms. CPU is fine at this scale.
Features Training: implicit feedback matrix (views, carts, purchases) — sparse matrix, low preprocessing. Inference: single user vector lookup + ANN search. Cold start: fall back to category popularity. Preprocessing complexity: LOW at inference, MEDIUM at training.
Peak QPS 20,000-50,000 on Black Friday (fires on every page load)Partial — depends on merchant size
Algolia hit "119,680 queries per second" on Black Friday 2025 across all clients. For a single $500M merchant, 20-50K page loads/sec at peak is plausible if total daily visits reach 5-10M (peak hour = 10-20x average). Queue-it shows Black Friday traffic is 2-3x a normal day overall, but second-by-second peaks can spike much higher.
Algolia — BFCM 2025: Peak Demand, Perfect Performance
. BUT: precompute top-20 recs for active users nightly → ~70-80% cache hit rate → real-time QPS drops to 5-15K.
Response time Target: <100 ms. Often loaded async, so 200-300 ms is tolerable without conversion impact. Quality matters more than speed — 10% relevance improvement = 2-5% GMV upliftSupported
McKinsey: "personalization most often drives 10-15% revenue lift" (range 5-25%). Barilliance: "product recommendations account for ~12-31% of e-commerce revenue." Amazon: ~35% of purchases attributed to recommendations. If recs drive ~15-30% of revenue, a 10% quality improvement translates to ~1.5-3% GMV uplift — the 2-5% claim is in the right range.
Barilliance — Personalized Product Recommendation Statistics
.
GDPR erasure MEDIUM-HIGH. Must delete: interaction logs, user embedding vector (Redis key delete), cached recs, impression/A/B test logs. Model: delete user's row from the ALS user matrix; full retraining on next weekly cycle naturally excludes them. Cross-system propagation to recommendation cache is the real work.
Data prep & training EC2 r6i.4xlarge (128 GB RAM) for ALS training via implicit library. GitHub Actions triggers training run, pulls interaction data from warehouse, trains ALS factors, builds FAISS IVF index, packages model.tar.gz (user/item factors + FAISS index) → uploads to S3. Precomputes nightly recs for top 100K users → writes to Redis. Tracked by MLflow.
Inference deployment Same Docker/ECS pattern as fraud. Container loads FAISS index + user factors from S3 artifact at startup. FastAPI /predict does in-process FAISS ANN lookup (~5 ms). FAISS is already optimized C++ — no ONNX needed. Redis caches precomputed recs for known users. ECS Fargate behind ALB, autoscaling 2-15 tasks. Same Dockerfile, different MODEL_TYPE=als_faiss.

C. Inventory / Demand Forecasting (Batch)

Dimension Assessment
Algorithm LightGBM global model (one model for all SKUs). 2,000-5,000 trees, ~200 features. Regression target: units sold in next 7/14/28 days. Beats Prophet on accuracy for e-commerceSupported
M5 Accuracy Competition (Walmart retail data): "All top-performing methods were pure ML methods that were significantly better than all statistical benchmarks. LightGBM was used by all of the top 50 competitors." Prophet was not among the top methods.
ScienceDirect — M5 Accuracy Competition: Results, Findings, and Conclusions
. Fallback for new SKUs: category-level moving average.
Model size 60-80 MB on disk. 120-300 MB in memory. Tiny by any standard.
Inference Batch only. Score 100K SKUs in ~30-120 seconds on a single CPU. Full pipeline (feature gen + scoring + write): 30-90 min.
Features ~150-250 features per (SKU, date). Lag features (sales 7d/14d/28d/1yr ago), rolling stats (7d/28d/90d mean/std/min/max), calendar features, promo flags, category, price tier. Preprocessing complexity: MEDIUM-HIGH — lag/rolling features require careful time-based windowing to avoid data leakage. Feature engineering is 60% of the work.
Peak QPS N/A — batch. Daily job: 100K SKUs. Weekly deep run: 500K+ SKUs. A single 16-vCPU / 64 GB node handles this easily.
Response time Batch SLA: results by 6 AM daily (before buying team starts). Missing the forecast → suboptimal replenishment → stockouts. 1% stockout rate on $500M GMV = ~$5M lost salesSupported
"Stock-outs cost retailers $1.2 trillion globally every year" (IHL Group). "69% of online shoppers shop with a competitor immediately when encountering out-of-stock items." For a $500M merchant: 1% stockout × $500M × 69% spill rate = ~$3.5M direct loss. With repeat customer attrition, $5M+ is a reasonable total impact estimate.
Mirakl — The $1.2T Problem: Why Out-of-Stocks Are Crushing Retailers
.
GDPR erasure LOW. SKU-level model, not user-level. Training data is aggregate sales counts per SKU per day — no individual user data. Deleting a user's orders from raw data could change aggregates, but negligibly at scale. No model retraining needed.
Data prep & training Feature engineering in SQL (warehouse-side). GitHub Actions triggers LightGBM training on EC2, pulls features as Parquet from S3. Exports ONNX model via onnxmltools.convert_lightgbm() for validation. Packages artifact → S3. Tracked by MLflow. Scheduled via Airflow DAG or cron.
Inference deployment No containers needed — batch only. Trained model runs as a scheduled Python script (Airflow or cron) that loads the S3 artifact, scores all SKUs, and produces a ranked operations digest — a prioritized purchasing plan that respects capital constraints and flags storage-fee risk. Operations opens a single report each morning and knows exactly what to order.

D. Customer Segmentation (Batch)

Dimension Assessment
Algorithm K-Means on standardized RFM + behavioral features, K=8-15 clustersSupported
RFM + K-Means is "one of the most famous, simple to utilize, and successful division strategies to enable advertisers to break down customer behavior." Published studies typically find K=4-8 optimal via elbow/silhouette analysis. K=8-15 is on the high end — most literature supports K=4-8 as more typical, though complex datasets with diverse customer bases can justify 10-12.
ResearchGate — Customer Segmentation Using RFM and K-Means Clustering
. PCA (50→15 components) before clustering handles correlation/scale. GMMs give soft assignments if needed. Skip DBSCAN — arbitrary shapes don't map to marketing personas.
Model size <1 MB. Centroids: 10 clusters × 50 features × 4 bytes = 2 KB. PCA matrix: 3 KB. Scaler: 400 bytes. The model is trivially small — it's just centroid coordinates.
Inference Batch. Assign 1M customers in <60 seconds (distance calculation). Full pipeline: 1-4 hours (warehouse query time dominates).
Features ~30-80 features. RFM core (recency, frequency, monetary), category preferences (% spend in 10-20 cats), device type, channel responsiveness, return rate. Must StandardScale all features (K-Means is distance-based). Log-transform monetary features. Preprocessing complexity: LOW-MEDIUM.
Peak QPS N/A — batch weekly/monthly. Output (user→segment) may be queried real-time as a simple Redis key-value lookup (~1 ms).
Response time Batch SLA: before next campaign send (T+24h is fine). Marketing is not latency-sensitive. Missing a weekly re-segmentation by a day has minimal impact.
GDPR erasure MEDIUM. Must delete: user's behavioral features, cluster assignment (user_id→segment_id). Model itself: centroids are group statistics — single user deletion doesn't change them. Cross-system propagation is the real challenge — segment assignments flow to CRM (Salesforce), customer engagement (Braze — a customer engagement platform for email, push, SMS), ad platforms. All must receive the deletion signal.
Data prep & training GitHub Actions daily schedule (4 AM UTC) + manual workflow_dispatch. Builds Docker container → pushes to ECR. RFM feature engineering in SQL, K-Means + PCA in scikit-learn. Model artifact (centroids + scaler + PCA) saved to S3. Tracked by MLflow.
Inference deployment GitHub Actions kicks off a one-shot ECS Fargate task (not a long-running service). Container runs the full pipeline: munge → train → score → generate HTML report → upload everything to S3. Outputs: report.html (shareable with marketing/sales via presigned URL), model.tar.gz, metrics.json. Also writes to segmentation/latest/ for stable URLs. Marketing receives the presigned link and is responsible for enacting campaigns on Braze, Salesforce, ad platforms, etc.

E. Dynamic Pricing (Batch)

Dimension Assessment
Algorithm Hybrid: ML demand elasticity (batch) + rule engine (real-time). Offline: LightGBM regression trained on historical price-demand data → outputs price elasticity per SKU. Online: Rule engine applies elasticity + constraints (price floor, ceiling, MAP, competitor parity). Optional: Thompson Sampling bandit for price point A/B testing. Do NOT start with pure RL — business constraints make it impracticalSupported
Industry consensus: pure RL for pricing requires massive exploration (showing "wrong" prices to learn), which is commercially unacceptable. Hybrid approaches — ML for elasticity estimation + rule engines for constraints — are the standard production pattern. Thompson Sampling bandits offer a safer exploration mechanism.
ScienceDirect — Dynamic Pricing with ML: Survey and Practice
.
Model size Elasticity LightGBM: 20-50 MB. Per-SKU coefficient table: ~8 MB in Redis (100K SKUs × 10 coefficients). Rule engine: stateless code, no model.
Inference Batch. Repricing worker runs every 10 minutes as a scheduled ECS task. Scores all SKUs (or priority subset), applies rule engine, writes final prices to Redis. Backend reads pre-computed price from Redis in 1-3 ms. Prices are at most ~10 minutes stale — acceptable for e-commerce.
Features ~80-120 for elasticity model. Demand velocity (rolling sales in 1h/6h/24h — same Redis pattern as fraud), inventory level, competitor price (requires scraping pipeline), promo flags, seasonality. Preprocessing complexity: MEDIUM. Same Redis velocity pattern as fraud — if you build it for fraud, reuse it here.
Peak QPS N/A — batch repricing. Price reads from Redis can handle 15-40K QPS on Black FridayPartial — same traffic caveat as fraud
Same reasoning as fraud QPS: depends on merchant size. Queue-it shows Black Friday traffic is 2-3x a normal day overall. For a $500M merchant with millions of daily page views, 15-40K page loads/sec at peak is the high end. Price lookups are cheaper than fraud scoring since they're Redis key-value reads.
Queue-it — 149 Black Friday Statistics (2026)
, but the repricing worker itself processes ~1-5K SKUs/cycle every 10 min.
Response time Batch SLA: prices refreshed every ≤10 minutes. Backend reads from Redis (~1-3 ms), so page load is never bottlenecked by pricing. 10-minute staleness is acceptable — consistency matters more than freshness: "price shock" (different price shown vs checkout) is the #1 driver of cart abandonmentPartial — broader than stated
Baymard Institute (50-study meta-analysis): "48% of US customers say unexpected costs are the main reason for abandoning shopping carts." The specific 15-25% figure for price-specific discrepancy is not directly sourced — the broader stat is 48% for all unexpected costs. With a 70% base abandonment rate, ~34% of all shoppers abandon due to cost surprises. Better framing: "Unexpected costs at checkout are the #1 reason for cart abandonment."
Baymard Institute — Cart Abandonment Rate Statistics (2026)
. The price in Redis is the price at checkout — no discrepancy.
GDPR erasure LOW-MEDIUM. Elasticity model is SKU-level, not user-level — no user data in the model. Must delete: user session/event logs (if demand velocity uses per-user signals), pricing impression logs (user_id saw price $X). No model retraining needed.
Data prep & training Monthly retrain: GitHub Actions triggers LightGBM training on EC2, exports ONNX via onnxmltools, packages model.tar.gz → S3. Daily coefficient refresh: Airflow DAG runs batch scoring of all SKUs, writes elasticity coefficients to Redis. Tracked by MLflow.
Inference deployment Two-layer architecture: pre-computed prices in Redis + background repricing worker. The e-commerce backend never calls the ML model directly — it reads the latest price from ElastiCache Redis (1-3 ms GET). A background repricing worker (ECS scheduled task) runs every 10 minutes, re-scores SKUs through the ONNX model + rule engine, fetches competitor prices (e.g. Prisync API), and writes updated prices back to Redis. Amazon adjusts prices ~2.5 million times per day — the average product price changes every ~10 minutes somewhere on the siteSupported
Amazon's dynamic pricing engine is widely reported to change prices 2.5M+ times daily across its catalog. CamelCamelCamel price tracking data shows frequent intra-day price fluctuations on popular products, often multiple times per hour during high-demand periods.
CamelCamelCamel — Amazon Price Tracker
. Repricing strategy: if the full catalog (100K SKUs) reprices in <10 minutes, just run them all every cycle. If it takes longer, set up a priority queue ranked by pageview activity — high-traffic SKUs reprice every cycle, long-tail SKUs reprice hourly/daily.

Design Principles

Principle What It Means
No vendor lock-in Every component is open-source or has drop-in replacements. FastAPI, Redis, MLflow, Airflow, Docker — all run anywhere. Swap your cloud provider without changing application code.
Shared base, per-model code All models share the same base Docker image (FastAPI + ONNX Runtime + common libs). Each model has its own Python module for data prep, feature queries, and database connectors — fraud pulls velocity features from Redis, reco queries FAISS + user vectors, pricing fetches competitor prices and inventory levels, etc. MODEL_TYPE env var selects which module runs. Same Dockerfile, different entrypoint logic.
Redis is the shared brain One Redis cluster serves all models that need fast lookups: velocity features (fraud), user embeddings + rec cache (reco), pre-computed prices (pricing). Build the Redis infrastructure once.
Batch is just a script Forecast, Segmentation, and Pricing don't need real-time serving — they're scheduled jobs. Pricing runs every 10 min, the others daily. Don't over-engineer what a scheduled task can handle.
MLflow is the registry Every model — from the trivial K-Means to the production XGBoost — registers in MLflow with versioning, metrics, and artifacts. One place to find any model.

Shared Infrastructure (build once)

Component Role Used By Cloud-Agnostic Options
Redis Feature store, cache, embeddings Fraud, Reco, Pricing Any managed Redis (ElastiCache, Memorystore, Azure Cache, Aiven) or self-hosted. Valkey as OSS drop-in.
FastAPI + ONNX Runtime + Docker Real-time model serving (optimized) Fraud, Reco (real-time); Pricing (batch worker reuses same image) ECS Fargate behind ALB. ONNX Runtime gives 2-4x speedup on tree models (XGBoost, LightGBM). FAISS stays native. Container portable to any runtime: K8s, Cloud Run, Fly.io.
MLflow Experiment tracking + model registry All 5 Self-hosted (Docker Compose) or managed (Databricks, MLflow on any VM). Object store backend (S3, GCS, Azure Blob, MinIO).
Data warehouse Training data + batch outputs All 5 Snowflake, BigQuery, Redshift, DuckDB, Postgres. The models don't care — they read DataFrames.
Airflow Batch orchestration Forecast, Segmentation, Pricing (all batch) Airflow, Prefect, Dagster, or plain cron. If ≤3 jobs, cron is fine. All are open-source.
GDPR deletion pipeline Cascade user data deletion All 5 Airflow DAG or any workflow engine. Fan out to Redis, warehouse, object store, CRM. No cloud-specific service required.

GDPR Deletion Architecture

Deletion Request (user_id)
    │
    ├── Redis: DEL user:{id}:* (features, embeddings, cached recs, coefficients)
    ├── Warehouse: DELETE FROM ... WHERE user_id = ? (training data, feature tables)
    ├── Object store: Mark for deletion in data catalog (training snapshots)
    ├── Prediction logs: DELETE WHERE user_id = ? (audit/monitoring tables)
    ├── Downstream: Push deletion to CRM, email, ad platforms via reverse ETL
    │
    └── Model retraining: NOT required per-request
        (next scheduled retrain naturally excludes deleted data)

SLA: All deletions complete within 72 hours of request
GDPR deadline: 30 calendar days
Buffer: 27 days for edge cases, manual review, legal holds

Deployment Order (Risk-Adjusted)

Order Model Rationale
1 Segmentation Lowest risk, batch, <1 MB model, no latency. Quick win for marketing. Validates warehouse → CRM pipeline.
2 Forecasting Batch, no user PII, no latency. Validates Airflow + LightGBM + MLflow pattern.
3 Fraud First real-time model. Highest business value. Deploy in shadow mode first (score but don't block). Validates Redis + FastAPI + Docker pattern that all real-time models reuse.
4 Pricing Batch repricing worker reuses fraud's Docker + Redis infrastructure. Rule engine needs business stakeholder alignment (takes time). 10-min cycle is non-critical — safe to iterate.
5 Reco Highest QPS, most complex caching. Deploy last so infrastructure is battle-tested.
