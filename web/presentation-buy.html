Campaign Configuration

  • Dataset group: Users, Items, Interactions
  • Recipe: USER_PERSONALIZATION (primary)
  • Fallback: POPULAR_ITEMS for cold-start users
  • Real-time events: PutEvents API for live behavior
  • Batch inference: Daily email recommendations
  • Filters: Exclude already-purchased, out-of-stock

Integration Pattern

Snowflake (interactions) → S3 → Personalize Dataset Import
↓

A/B Testing Strategy

Test Plan

  • Control: Current recommendation logic
  • Variant A: USER_PERSONALIZATION recipe
  • Variant B: SIMILAR_ITEMS recipe
  • Split: 33% / 33% / 34%
  • KPIs: CTR, conversion rate, AOV, revenue/session
  • Duration: 2-4 weeks for statistical significance

Expected Metrics

  • Click-through rate: +15-30% vs baseline
  • Conversion rate: +5-12%
  • Average order value: +8-15%
  • Overall revenue uplift: 8-15%
Amazon Personalize uses three dataset types: users, items, and interactions. We recommend the USER_PERSONALIZATION recipe for the primary campaign with a POPULAR_ITEMS fallback for cold-start users. Real-time events are captured via the PutEvents API so recommendations improve as users browse. We apply filters to exclude already-purchased and out-of-stock items. For A/B testing, we run three variants — your current logic as control, USER_PERSONALIZATION, and SIMILAR_ITEMS — measuring click-through rate, conversion, and average order value over 2-4 weeks.
APPENDIX A3

AWS Fraud Detector Deep-Dive

Detector Configuration

  • Detector type: Online Fraud Insights (OFI)
  • Training data: 6+ months historical transactions
  • Variables: IP, email, device, amount, velocity
  • Model type: Pre-trained + custom fine-tuned
  • Scoring: 0-1000 risk score per transaction

Rules Engine

Custom Business Rules

  • Auto-approve: Score < 200, known customer, same device
  • Review queue: Score 200-700, flag for manual review
  • Auto-block: Score > 700, or velocity > 5 txns/min
  • Amount guardrail: Block all > $5,000 without 2FA

Real-Time Flow

Transaction → API Gateway
↓
GetEventPrediction API
↓
Fraud Detector Score + Rules
↓
Outcome: APPROVE | REVIEW | BLOCK
↓
Response → Payments API (<100ms)

Performance Targets

  • Precision: >95% (minimize false positives)
  • Recall: >85% (catch most real fraud)
  • Latency: <100ms p99
  • Cost: ~$0.07 per event evaluation
AWS Fraud Detector uses the Online Fraud Insights model type, which is pre-trained on Amazon's own fraud data and then fine-tuned on your historical transactions. It produces a 0-1000 risk score per transaction. We layer custom business rules on top: auto-approve low-risk transactions, queue medium-risk for manual review, and auto-block high-risk. The entire flow — from transaction event to approve/block decision — runs in under 100 milliseconds. At 7 cents per event evaluation, this is dramatically cheaper than building and maintaining a custom fraud model.

  • Error handling: DLQ + SNS alerts + retry policies

Snowflake as Integration Hub

  • All prediction results land in Snowflake tables
  • External functions call AWS APIs from SQL
  • Streams detect new predictions for downstream triggers
  • Data sharing for cross-team access
  • Single audit trail across all services

Event Flow Architecture

  • Web events: Client SDK → Kinesis → Personalize PutEvents
  • Transactions: Payments → API GW → Fraud Detector
  • Feedback loop: Outcomes → Snowflake → retrain signals
  • Monitoring: CloudWatch metrics → SNS → PagerDuty
Integration is where the rubber meets the road. Real-time services expose simple API contracts: fraud evaluation returns a score and outcome, recommendations return ranked items, pricing returns a suggested price with confidence and guardrail flags. A key advantage of managed services: AWS handles infrastructure security, patching, DDoS protection, and SOC2 compliance for the service layer.
