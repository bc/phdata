APPENDIX A1

Detailed Architecture

This is the full infrastructure diagram. Everything runs within a VPC with public and private subnets. SageMaker endpoints run in the private compute subnet with auto-scaling from 2 to 20 instances. Snowflake connects via PrivateLink for security. Redis provides caching for the recommendation engine. Here's the detail on the real-time pipelines. For fraud detection, a transaction event hits Kinesis, Lambda enriches it with features from DynamoDB, calls the SageMaker endpoint, and returns approve/flag/block — all under 100 milliseconds. For recommendations, we use Redis caching aggressively. On cache miss, we call SageMaker with an A/B split between model versions. If the endpoint times out, we gracefully fall back to popular items from cache.

Error Handling & Retry Strategy

Step Functions retry with exponential backoff (3 attempts, max 5 min delay). On final failure: SNS alert → PagerDuty → on-call engineer. Previous successful output remains active until new run succeeds.

The batch pipeline uses EventBridge cron triggers to kick off Step Functions. Data flows from Snowflake to S3, through SageMaker Batch Transform, gets validated, and results land back in Snowflake. The Step Function handles retries with exponential backoff — if all retries fail, we alert via PagerDuty and keep the previous successful output active. Customer segmentation runs weekly, inventory forecasting runs daily.

Monitoring uses SageMaker Model Monitor for data drift and prediction drift, plus CloudWatch for operational metrics. When drift exceeds thresholds — PSI above 0.2, for example — it triggers an automated retraining pipeline. The new model runs as a "challenger" against the current "champion" on shadow traffic. If the challenger performs better on the holdout set, it auto-promotes. This means your models stay fresh without manual intervention.

We have two CI/CD pipelines. The model pipeline handles ML code changes — it runs unit tests, validates model accuracy against gates, deploys to staging for integration tests, then does canary rollout in production. If metrics degrade during canary, it auto-rolls back. The infrastructure pipeline handles Terraform changes — plan on PR, security scan, apply to staging on merge, manual approval for production. Terraform state is managed in S3 with DynamoDB locking.
APPENDIX A6

Security & GDPR Compliance

Encryption

  • At rest: KMS-managed keys for S3, EBS, RDS, DynamoDB
  • In transit: TLS 1.3 everywhere, VPC endpoints
  • Snowflake: PrivateLink, no public internet exposure
  • Model artifacts: Encrypted in S3, signed with KMS

Access Control

  • IAM: Least-privilege roles per service
  • SageMaker: Execution roles scoped to specific models
  • Snowflake: Row-level security for PII tables
  • MFA: Required for all human access

GDPR Compliance

  • PII masking: Tokenization in feature pipelines
  • Right to erasure: Automated deletion pipeline
  • Data retention: Configurable per data class
  • Consent tracking: Integration with consent management

Audit & Logging

  • CloudTrail: All API calls logged
  • VPC Flow Logs: Network traffic audit
  • Model predictions: Logged for explainability
  • Access reviews: Quarterly IAM audit
Security and GDPR are first-class concerns, not afterthoughts. Everything is encrypted at rest with KMS and in transit with TLS 1.3. Snowflake connects via PrivateLink — no data traverses the public internet. For GDPR, we implement PII tokenization in the feature pipelines, an automated right-to-erasure pipeline, and configurable data retention policies. All access is logged in CloudTrail, and we do quarterly IAM audits.

APPENDIX A8

Feature Store Design

Online Store (Real-Time)

SageMaker Feature Store + DynamoDB

  • Latency: <10ms single-digit ms lookups
  • Use case: Fraud features (transaction velocity, device fingerprint, account age)
  • Use case: Reco features (user embeddings, recent behavior)
  • Update: Streaming via Kinesis → Lambda → Feature Store
  • TTL: Feature-specific (1hr to 30 days)

Offline Store (Batch)

S3 + Snowflake

  • Format: Parquet on S3, tables in Snowflake
  • Use case: Segmentation features (12-month aggregates)
  • Use case: Inventory features (historical sales, seasonality)
  • Training: Point-in-time correct feature retrieval
  • Versioning: Feature group versions in SageMaker FS
