Why XGBoost

Interpretability

SHAP values explain every fraud decision. Compliance team can audit any prediction. Feature importance identifies key fraud signals.

Speed

XGBoost on Triton achieves <5ms model inference. With feature lookup and rule engine, total pipeline stays under 50ms.

Rule Engine Overlay

Business rules run as Triton pre-processing step: velocity limits (10+ txns/min → auto-block), known device blocklists, geo-velocity checks. Rules bypass ML when signal is clear.

The fraud detection pipeline is where XGBoost really shines. The model runs on Triton Inference Server as part of an ensemble — the rule engine runs as a pre-processing step in the same Triton graph. This keeps everything in-process for sub-50ms total latency. XGBoost gives us two things a neural network can't: first, SHAP values that explain every single fraud decision for your compliance team. Second, feature importance that tells you exactly which signals are driving fraud. The rule engine handles obvious cases — 10 transactions per minute from the same card gets auto-blocked without calling the model.
APPENDIX A3

Algorithm Deep-Dive: Recommendations

Two-Tower + FAISS + LightGBM

Pipeline Details

Candidate Generation (Two-Tower)

  • User tower: browsing history, purchase history, demographics
  • Item tower: product attributes, descriptions, images
  • Both produce 128-dimension embeddings
  • Trained on implicit feedback (clicks, purchases)

FAISS Approximate Nearest Neighbor

  • IVF-PQ index over all product embeddings
  • Returns top 100 candidates in <10ms
  • Index rebuilt nightly, updated hourly for new products

LightGBM Re-Ranking

  • Features: similarity score, price, stock, recency, user segment
  • Business rules: diversity, margin targets, sponsored items
  • A/B testing: compare tower versions + ranking strategies
The recommendation pipeline has three stages. First, the two-tower neural network produces embeddings — a user tower that encodes browsing and purchase behavior, and an item tower that encodes product attributes. Both produce 128-dimensional vectors. Second, FAISS does approximate nearest neighbor search to find the top 100 candidate items in under 10 milliseconds. Third, LightGBM re-ranks those candidates using context features like price, stock level, and user segment, plus business rules for diversity and sponsored items. Total pipeline under 150 milliseconds. The beauty is each stage can be improved independently.

  • Right to erasure: Automated deletion pipeline
  • Data retention: Configurable per data class
  • Consent tracking: Integration with consent management
  • SHAP audit trail: Explain any ML decision

Audit & Logging

  • CloudTrail: All AWS API calls logged
  • K8s Audit: All API server events logged
  • Model predictions: Logged for explainability
  • Access reviews: Quarterly IAM + RBAC audit
  • Compliance reports: Automated via OPA
Security for a Kubernetes-native platform has additional layers compared to a managed-service approach. OPA Gatekeeper enforces pod security policies as code — no container can run as root, no privileged containers allowed. IRSA gives each pod its own IAM role, so there are no shared credentials floating around. Calico network policies enforce deny-by-default — pods can only communicate on explicitly allowed paths. Secrets are managed by the External Secrets Operator pulling from AWS Secrets Manager. For GDPR, we add SHAP audit trails so every ML prediction can be explained — critical for GDPR's right to explanation.
